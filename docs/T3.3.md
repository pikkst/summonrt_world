# Task T3.3 – Convert `captureCreature` to Timed Mission

## What was found

`captureCreature` in `src/stores/game/modules/missionModule.ts` performed creature capture as an immediate action with inline rarity/affinity/skill calculations followed by an instant success/failure outcome. This meant:

- The entire capture sequence (element affinity gate, creature generation, roll, quest update/combat trigger) executed synchronously in a single Zustand action.
- The mission queue infrastructure built in Sprint 2 was completely bypassed for capture attempts.
- The 1-second heartbeat had no `CAPTURE_CREATURE` callback, so there was no way to resolve captures through the unified time-based system.
- There was no UI feedback for the 60-second capture window — the result was shown instantly.

## What was done

### 1. Added `CAPTURE_CREATURE` mission type

Added `'CAPTURE_CREATURE'` to the `MissionType` union in `src/core/missionQueue.ts` so the heartbeat can track and resolve capture attempts through the mission queue.

### 2. Extended `HeartbeatCallbacks`

Added `CAPTURE_CREATURE: (mission: ActiveMission) => void` to the `resolveMissionCallbacks` interface in `src/core/heartbeat.ts`.

### 3. Added `capturing` state to GameStoreState

Added a new `capturing` state property (mirroring the existing `searching` pattern) to track active capture missions:

```typescript
capturing: {
  missionId: string;
  creature: {
    key: string;
    name: string;
    class: CreatureClass;
    type: CreatureType;
    elements: Element[];
    baseHealth: number;
    baseAttack: number;
    baseDefense: number;
    baseSpeed: number;
    baseMana: number;
    baseExpValue: number;
    skills: { key: string; name: string; description: string; element?: Element; power: number; cost: number }[];
    description: string;
    isBoss?: boolean;
  };
  endTime: number;
  totalDuration: number;
} | null;
```

The generated `CreatureTemplate` is serialized into this state so the exact target creature is preserved for the duration of the 60-second attempt.

### 4. Converted `captureCreature` to mission-based

`captureCreature` is now a timed mission action:

- Checks for an existing `capturing` state to prevent concurrent captures.
- Validates party capacity (max 6 creatures), world existence, and element affinity exactly as before — these remain hard gates at mission start.
- Generates a `CreatureTemplate` using the existing seeded RNG and stores it in the `capturing` state.
- Creates a `CAPTURE_CREATURE` mission with a fixed 60-second duration via `addMissionWithModifiers`.
- Logs the start of the capture ritual with the ETA.

### 5. Added `finishCapture` callback

New `finishCapture` action that resolves the timed capture attempt:

- Recalculates capture probability at resolution time using current player state (skills like `beastmaster`, `wild_whisper`, `nebula_shroud` may have changed during the attempt).
- Re-evaluates level gating, rarity penalties, and level differences at resolution time.
- Performs the success/failure roll:
  - **Success**: appends the new `CreatureInstance` to `player.creatures` using immutable state updates, increments summon quest progress, logs the victory.
  - **Failure**: logs the failure with the capture chance percentage, and if the player has at least one creature, starts combat against the wild creature after a 500ms delay.
- Clears the `capturing` state in all branches.

### 6. Added heartbeat resolution callbacks

Both `processOfflineCatchUp` and `startHeartbeat` now include:

```typescript
CAPTURE_CREATURE: () => {
  const state = get();
  if (state.capturing) {
    state.finishCapture();
  }
},
```

This ensures captures resolve during live gameplay and after offline periods.

### 7. Persisted `capturing` state in save/load

- Added `capturing: state.capturing` to the save payload in `playerModule.ts`.
- Added `capturing: data.capturing || null` to the load logic so capture attempts survive page reloads and offline periods.

### 8. Added `capturing: null` to initial state

Initialized `capturing: null` in `gameStore.ts` `initialGameState`.

### 9. Updated UI for capture progress

In `src/ui/GameShell.tsx`:

- Added a `capturing` state selector and `captureProgress` counter with a 100ms interval.
- Added a progress bar overlay in the Interactive Points section when a capture is active, showing "Binding Soul..." with a percentage counter.
- Disabled the "Manifest Soul" button and added a warning log when the player attempts to capture while already attempting one.
- Updated the command-line handler to warn the player if they try to capture during an active attempt.

### 10. Added tests

- **`src/test/missionQueue.test.ts`**: Added `CAPTURE_CREATURE` to the `MISSION_TYPES` property-based test array; added a specific test verifying `CAPTURE_CREATURE` mission creation and minimum duration enforcement.
- **`src/test/heartbeat.test.ts`**: Added `CAPTURE_CREATURE: vi.fn()` to every `resolveMissionCallbacks` mock object across all test cases (createHeartbeat, offline catch-up, resolveMission, updateMissionStatus, processMissionResult).

## Behavioral Changes

| Aspect | Before | After |
|--------|--------|-------|
| Capture action | Immediate generation, roll, and result | 60-second timed mission enqueued and resolved by heartbeat |
| Element affinity check | Synchronous gate before roll | Hard gate at mission start; final roll occurs after 60s |
| Capture probability | Fixed at action time | Recalculated at mission resolution using current skills/level |
| Mission queue | Bypassed entirely for capture | `CAPTURE_CREATURE` missions flow through the same queue as exploration, search, and gather |
| Offline catch-up | No capture resolution | Offline period resolves pending capture attempts |
| Save/load | No capture persistence | `capturing` state persisted, survives reloads |
