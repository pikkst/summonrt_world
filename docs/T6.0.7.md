## Summary
Implemented the Demonlord challenge issuance and acceptance system, allowing players to challenge the Demonlord on floors 50-100 and enabling the Demonlord to accept challenges for combat.

## What Was Done

### 1. Core Demonlord Challenge Functions (`src/core/demonlord.ts`)

#### `issueChallenge(state, challengerId, challengerName, currentTime)`
- Adds a challenger to the `pendingChallenges` queue
- Prevents duplicate challenges from the same player
- Returns updated state with new challenge appended

#### `acceptChallenge(state, currentTime)`
- Moves the first pending challenge to `activeChallenge`
- Removes accepted challenge from `pendingChallenges` queue
- Triggers combat preparation log message

#### `resolveChallenge(state, victorId, currentTime)`
- Determines winner of Demonlord combat
- If player wins: Sets player as new Demonlord, awards +5 passive points, records defeat timestamp
- If Demonlord wins: Clears active challenge, maintains current Demonlord status
- Updates `currentLordPlayerId` and `currentLordPlayerName` appropriately

### 2. Store Module Integration (`src/stores/game/modules/demonlordModule.ts`)

#### `challengeDemonlord()`
- Validates player is on Demonlord floors (50+)
- Calls core `issueChallenge` function
- Adds system log entry for challenge issuance

#### `acceptDemonlordChallenge()`
- Validates pending challenges exist
- Calls core `acceptChallenge` function
- Logs challenge acceptance for UI feedback

#### `resolveDemonlordCombat(victorId: string)`
- Identifies victor and updates state accordingly
- Awards passive points to winning player
- Logs victory/defeat outcome

### 3. Type System Support (`src/types/game.ts`)

Added types for the challenge system:
- `pendingChallenges: Array<{ challengerId, challengerName, issuedAt }>` - Challenge queue
- `activeChallenge?: { challengerId, challengerName, acceptedAt }` - Current combat challenge
- `defeatedAt?: number` - Timestamp of last defeat

### 4. Mission Queue Integration (`src/core/missionQueue.ts`)

Added `DEMONLORD_ENCOUNTER` to `MissionType` enum for integration with timed mission system.

## Design Rationale

### Challenge Flow
The system implements a three-stage flow:
1. Player issues challenge â†’ Added to pending queue
2. Demonlord accepts â†’ Moves to active challenge, combat begins
3. Combat resolves â†’ Winner determined, state updated

### FIFO Challenge Queue
Uses a first-in-first-out queue for `pendingChallenges` to ensure fair challenge ordering when multiple players attempt to challenge simultaneously.

### Influence-Based Bonuses
The Demonlord's influence (0-100) affects combat:
- Higher influence = Demonlord damage bonus +50% at full influence
- Higher influence = Reduced player defense -30% at full influence
- Career tree bonuses are applied multiplicatively on top
