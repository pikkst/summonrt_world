# Task T3.7 – Unify `GameEngine.tickCreatures()` XP granting for entire party on mission completion

## What was found

The project’s XP logic was scattered across multiple store modules with no single source of truth:

- `GameEngine.tickCreatures()` in `src/core/gameEngine.ts` was **dead code** — never instantiated or called from the live game loop. It only mutated `+1 XP` to the player and never touched creatures.
- Creature XP was handled **ad-hoc in three separate places**:
  - `combatModule.ts` `handleVictory` had an inline level-up loop with `newExp >= newLevel * 100` thresholds, stat gains, and skill-learning logic.
  - `playerModule.ts` and `missionModule.ts` `finishActivity` only gave **player XP** (`creatureXp / 4`). The displayed creature XP message was never applied to the `CreatureInstance` — a silent bug.
- Mission resolution callbacks in `heartbeat.ts` (`processOfflineCatchUp` and `startHeartbeat`) resolved missions but **never granted any XP** to the party. `assigned_creatures` on missions was unused for XP.
- `getWorldModifier` was imported from `../helpers.ts` in three modules, but `helpers.ts` never exported it — a pre-existing type-error hidden by loose test runner config.

This meant the core gameplay loop (explore → gather → summon → combat) produced no party progression on mission completion, and the existing `GameEngine` class — the designated “Decoupled Core” from T0.13 — was completely disconnected from runtime state.

## What was done

### 1. Centralized creature XP functions in `xpCurve.ts`

Added two exported helpers so all XP math flows through one module:

- **`applyCreatureXP(creature, xp, maxLevel = 1000)`** — accumulates XP, runs the level-up threshold check (using the canonical `getXPThreshold` curve), increments stats (HP/MP/ATK/DEF/SPD), and returns a fully updated `CreatureInstance`. Preserves `currentHealth` / `currentMana` when no level-up occurs so concurrent healing is not lost.
- **`grantPartyXP(creatures, creatureIds, baseXP, maxLevel)`** — distributes `baseXP` equally across matching `creatureIds`, returns the updated creature array plus a list of IDs that leveled up.

`src/core/xpCurve.ts`

### 2. Refactored `GameEngine` to return immutable state patches

Updated `GameEngine` so it no longer mutates its constructor argument directly:

- `processTurn()` now returns `Partial<GameEngineState>` by merging patches from `tickWorlds()`, `tickPlayer()`, and `tickCreatures()`.
- `tickWorlds()` tiles are copied into a new `Map` to keep updates immutable.
- `tickPlayer()` creature health/mana regeneration uses each creature’s actual stored `maxHealth` / `maxMana` instead of freshly generated template values (which previously drifted randomly every tick).
- `tickCreatures()` now uses `getXPThreshold` for player level arithmetic (bigint-safe) and calls `applyCreatureXP(c, 1)` for **every creature** in the party, unifying turn-based and mission-completion XP under the same formula.

Added a new public method:

- **`grantPartyXP(creatureIds, baseXP)`** — instantiates the centralized `grantPartyXP` helper and returns a `{ player: { creatures: [...] } }` state patch suitable for direct use in Zustand `set()` calls.

`src/core/gameEngine.ts`

### 3. Wired unified party XP into mission resolution

Changed both heartbeat initializers (`processOfflineCatchUp` and `startHeartbeat`) so every `resolveMissionCallbacks` handler grants party XP after resolving its primary action:

- Each callback now accepts the `ActiveMission` parameter.
- A local `getBaseXP(mission)` helper computes a tier-aware base XP (`15–40` depending on type) scaled by `world_layer`.
- After the mission’s state-changing callback (e.g. `finishMovement`, `finishSearch`, `finishCapture`) runs, `state.grantMissionXP(...)` is called with the full party’s creature IDs and the computed base XP.
- `grantMissionXP` uses a functional `set((state) => ...)` update, ensuring it always reads the latest state (including any concurrent healing from `finishMovement`) and applies XP on top without overwriting unrelated fields.

`src/stores/game/modules/missionModule.ts`

### 4. Added `grantMissionXP` to the store interface

- Extended `GameActions` in `src/stores/game/types.ts` with `grantMissionXP(creatureIds, baseXP)`.
- Implemented the action in `missionModule.ts` using the unified `grantPartyXP` function.
- Re-exported `getWorldModifier` from `helpers.ts` to fix the pre-existing import error without changing call sites.

### 5. Unified combat victory XP in `combatModule.ts`

Replaced the inline creature XP loop in `handleVictory` with `applyCreatureXP`, preserving the existing skill-learning and MAX_LEVEL logic while removing duplicated threshold arithmetic. This keeps combat level-ups consistent with training and mission XP.

`src/stores/game/modules/combatModule.ts`

### 6. Fixed `finishActivity` creature XP in both store modules

Both `playerModule.ts` and `missionModule.ts` `finishActivity` handlers for `creature_training` now:

- Call `applyCreatureXP(creature, creatureXp)` to actually update the creature’s experience.
- Log level-up stats (HP/ATK/DEF/SPD) when training triggers a level-up.
- Use a functional `set()` to merge the updated creature back into the player state without losing concurrent changes.

## Behavioral Changes

| Aspect | Before | After |
|--------|--------|-------|
| `GameEngine` | Dead code, never called | Returns immutable state patches; `tickCreatures()` distributes XP to entire party |
| Creature XP on missions | None — callbacks were stubs or only updated player state | Every mission type grants scaled XP to all creatures via unified `grantPartyXP` |
| Combat victory XP | Inline `while` loop with `newLevel * 100` thresholds | `applyCreatureXP` using canonical `getXPThreshold` curve |
| Training activity | Logged creature XP but never updated `CreatureInstance` | `applyCreatureXP` actually writes experience, handles level-ups |
| `getWorldModifier` | Imported from wrong module (`helpers.ts`), causing TS error | Re-exported from `helpers.ts` for correct resolution |
| State updates | Direct mutation inside `GameEngine`; mixed `set()` patterns | Functional `set((state) => ...)` patches everywhere, safe against concurrent updates |
