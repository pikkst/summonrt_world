# Task T3.2 – Convert `searchArea`/`gatherResource` to Timed Missions

## What was found

`searchArea` and `gatherResource` in `src/stores/game/modules/missionModule.ts` performed resource discovery and gathering as immediate actions:

- `searchArea` immediately checked if the current tile had a resource, found 1-2 units, and added them to inventory without any time delay.
- `gatherResource` immediately gathered 1-3 units of the specified resource if available on the tile.
- These immediate actions bypassed the mission queue infrastructure built in Sprint 2.
- The 1-second heartbeat loop was already active (from T3.1) but had no callbacks registered for SEARCH_AREA or GATHER_RESOURCE mission types.

## What was done

### 1. Added new mission types to missionQueue.ts

Added `SEARCH_AREA` and `GATHER_RESOURCE` to the `MissionType` union type:

```typescript
export type MissionType =
  | 'EXPLORE_TIER_1'
  | 'SCOUT_DUNGEON'
  | 'SMELT_ORE'
  | 'CRAFT_ITEM'
  | 'STORE_VISIT'
  | 'TAX_EDICT'
  | 'CARAVAN_ROUTE'
  | 'SEARCH_AREA'
  | 'GATHER_RESOURCE';
```

Also extended `MissionModifiers` to include `resource_type?: string` for tracking which resource is being gathered.

### 2. Added searching state to GameStoreState

Added a new `searching` state property to track active search/gather missions:

```typescript
searching: {
  missionId: string;
  resourceType?: string;
  endTime: number;
  totalDuration: number;
} | null;
```

### 3. Converted `searchArea` to mission-based

- Checks for existing `searching` state to prevent concurrent operations.
- Creates a `SEARCH_AREA` mission with duration 15-45 seconds (randomized).
- Stores the mission ID, resource type, and timing in the `searching` state.
- Logs the start of the search operation with duration.

### 4. Converted `gatherResource` to mission-based

- Checks for existing `searching` state to prevent concurrent operations.
- Creates a `GATHER_RESOURCE` mission with duration 15-45 seconds.
- Stores resource type in mission modifiers for resolution.
- Logs the start of the gathering operation with duration.

### 5. Added `finishSearch` callback

New action that:
- Reads the tile at the player's current position.
- Calculates gathered amount (1-2 units, based on remaining resource).
- Updates player inventory and tile resource quantity.
- Logs the result of the search.
- Clears the `searching` state.

### 6. Added heartbeat resolution callbacks

Both `startHeartbeat` and `processOfflineCatchUp` now include:

```typescript
SEARCH_AREA: () => {
  const state = get();
  if (state.searching) {
    state.finishSearch(state.searching.resourceType!);
  }
},
GATHER_RESOURCE: (mission: ActiveMission) => {
  const state = get();
  const resourceType = mission.modifiers?.resource_type as string | undefined;
  if (state.searching && resourceType) {
    state.finishSearch(resourceType);
  }
},
```

### 7. Persisted searching state in save/load

Added `searching: state.searching` to save payload in `playerModule.ts` and `searching: data.searching || null` to load logic. Also added `searching: null` to initial game state in `gameStore.ts`.

### 8. Added `finishSearch` to GameActions interface

Added the new `finishSearch` method signature to the `GameActions` interface in `types.ts`.

### 9. Added tests for new mission types

Added tests in `missionQueue.test.ts`:
- Verifies `SEARCH_AREA` and `GATHER_RESOURCE` mission creation with correct types.
- Verifies `resource_type` modifier persistence.
- Verifies speed modifier compression affects mission duration.

## Behavioral Changes

| Aspect | Before | After |
|--------|--------|-------|
| `searchArea` | Immediately found resources, no delay | Creates 15-45s mission, resolves via heartbeat |
| `gatherResource` | Immediately gathered resources, no delay | Creates 15-45s mission, resolves via heartbeat |
| Mission queue | Only used for exploration | Now handles search and gather missions too |
| Resource timing | Instant gratification | Strategic timing with XP rewards |
