# T1.11 – Add career_category filter UI component (show only node category relevant to player)

## Findings

The `SkillTreePanel.tsx` component rendered all 19 career tree nodes in a single flat view with no mechanism to narrow focus by career category. For players with multiple unlocked career paths, this became visually noisy and made it difficult to plan progression in a specific sector.

### Existing behavior
- `SkillTreePanel` pulled `Object.values(getAllNodes())` and rendered every node unconditionally
- Connections (SVG lines between nodes) were computed over the full node list
- No category-based filtering or relevance indicators existed
- Player archetype was not mapped to career categories at the data level

### Relevant data available
- `CareerNode.career_category` values in `nodes.json`: `Blacksmith`, `Explorer`, `Official`, `Shopkeeper`, `Broker`, `Summoner`, `General`
- `PlayerState.archetype` exists but has no 1:1 mapping to career categories in the current data layer
- `PlayerState.unlocked_node_ids` tracks which nodes the player has already invested in

## Design Decisions

1. **Filter default**: "All" — the safest default because a new player has only `root_hub` (General) unlocked and would otherwise see an empty tree on first open.
2. **Relevance signal**: A small green dot indicator on category buttons where the player has at least one unlocked node (excluding `root_hub`). This satisfies "show only node category relevant to player" by highlighting categories the player has already engaged with.
3. **Root hub visibility**: `root_hub` (General category) is always rendered regardless of active filter, preserving the central connection point.
4. **Connection scoping**: SVG connection lines are only drawn when both endpoints exist in the currently filtered node set, preventing orphaned lines.

## Changes Made

### `src/ui/SkillTreePanel.tsx`
- Added `CATEGORIES` constant for the six non-General career categories
- Added `activeCategory` state (`CareerCategory | 'All'`) defaulting to `'All'`
- Added `relevantCategories` memo: derives which career categories have at least one unlocked node (excluding `root_hub`)
- Added `filteredNodeList` memo: filters nodes by `activeCategory`, always including `root_hub`
- Added `filteredIds` Set for O(1) endpoint lookups in connection rendering
- Updated connection loop to only render edges where both nodes are in `filteredNodeList`
- Updated node render loop to iterate over `filteredNodeList`
- Added filter bar in the header with:
  - "ALL" toggle button
  - One button per career category with uppercase tracking-widest styling matching the existing dark theme
  - Active state uses `bg-indigo-500/20 border-indigo-500 text-indigo-400`
  - Inactive state uses `bg-black/40 border-gray-800 text-gray-500`
  - Green emerald-500 dot indicator on categories present in `relevantCategories`

### Filter UI placement
Located directly below the header row containing the Points counter, Respec button, and Close button. Uses `overflow-x-auto` to remain usable on narrower screens.

## Technical Details

- Category filtering is purely client-side and derived from existing Zustand state; no backend changes required
- Type imports updated to include `CareerCategory` from `../data/careerTree/types`
- Unused `canUnlockNode` import retained for future use (imported in original file)

- `npm run test` passes (7/7 careerTree tests)
- Visual behavior verified: filter buttons render 7 options (All + 6 categories), green relevance dots appear only for categories with unlocked nodes
