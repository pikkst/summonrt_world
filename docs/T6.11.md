# T6.11 – Add Trap/Puzzle Minigame UI (Text-Based Choices)

## Summary
Added a dedicated text-based minigame UI for trap and puzzle rooms in the dungeon system. The UI appears directly within the dungeon descent flow, presenting atmospheric descriptions and clear choice buttons that integrate with the existing trap and puzzle resolution logic.

## What Was Found

### Existing Implementation
The backend logic for trap and puzzle rooms was already implemented in `src/core/dungeonGenerator.ts`:
- `generateTrapInteraction(rng)` — generates trap variants (dart trap, toxic gas, hidden blade) with choice sets
- `generatePuzzleInteraction(rng, worldIndex)` — generates puzzle variants (elemental pedestals, rune sequence, celestial lock) with choice sets
- `resolveTrapRoom(choice, playerSpeed, playerDexterity, playerDefense, rng)` — resolves trap choices with agility/defense checks
- `resolvePuzzleRoom(choice, rng)` — resolves puzzle choices with 35% chance fallback on wrong answers

The dungeon descent flow in `src/stores/game/modules/missionModule.ts` (`descendDungeon`) already:
- Detects trap/puzzle room types during floor descent
- Creates `RoomInteractionState` with choices and messages
- Sets `combat.active = false` and `combat.roomInteraction.active = true`
- Exposes `resolveTrapRoom` and `resolvePuzzleRoom` actions in `src/stores/game/modules/playerModule.ts`

### Gap Identified
Although the room interaction UI existed in `CombatPanel` (inside `src/ui/GameShell.tsx`), it only rendered when `screen === 'combat'`. The dungeon flow keeps `screen === 'dungeon'` for trap/puzzle encounters, meaning players never saw the trap/puzzle minigame choices during normal dungeon descent. The DungeonPanel only displayed a generic "Proceed Carefully" button, bypassing the minigame entirely.

## What Was Done

### Changes to `src/ui/GameShell.tsx`

#### DungeonPanel trap/puzzle minigame UI
Added a dedicated trap/puzzle minigame section in `DungeonPanel` that renders when `combat.roomInteraction.active` is true and the room type is `'trap'` or `'puzzle'`:

- **Header** — Shows hazard/puzzle icon and room title ("Hazard Encounter" or "Puzzle Room")
- **Narrative description** — Displays the generated trap/puzzle flavor text in a styled container
- **Choice grid** — Renders 3 clickable choice buttons, each with a bold label and a descriptive subtitle
- **Result feedback** — Shows success or failure message after resolution, plus damage taken for traps
- **Continue button** — Allows the player to proceed with descent after resolving the encounter

#### Store bindings
Bound `resolveTrapRoom` and `resolvePuzzleRoom` from the game store so the DungeonPanel choice buttons can trigger resolution directly.

## Implementation Details

### Trap Room Minigame
- Presents the trap description (e.g., dart trap, toxic gas, hidden blade)
- Offers 3 choices: evasion/maneuver (dodge, duck, hold breath, rush), defense (block, parry), or endure
- Resolution uses player `speed`, `dexterity`, and `defense` stats via `resolveTrapRoom`
- Failed attempts deal 8–12 damage; successful evasion takes none

### Puzzle Room Minigame
- Presents the puzzle riddle (elemental order, rune sequence, celestial lock)
- Offers 3 labeled choices with flavor descriptions
- Resolution uses `resolvePuzzleRoom` which checks correctness and applies a 35% success fallback for wrong guesses
- Higher world indices add descriptive flavor text ("The magic here feels stronger…")

### Result States
- **Success** — Green-highlighted feedback card with the success message
- **Failure** — Red-highlighted feedback card with the failure message and damage taken (traps only)
- **Continue Descent** — Button advances the dungeon run to the next floor
