# Task T3.1 – Convert `movePlayer` to Non-Immediate Exploration Missions

## What was found

`movePlayer` in `src/stores/game/modules/missionModule.ts` performed exploration as an immediate action with a `setTimeout` callback that directly invoked `finishMovement`. This meant:

- Movement was tightly coupled to the browser timer loop.
- The mission queue and heartbeat infrastructure (built in Sprint 2) were bypassed during normal gameplay; the heartbeat was only used inside `processOfflineCatchUp` where all `resolveMissionCallbacks` were empty stubs.
- `createActiveMission` initialized missions with `status: 'PENDING'`, but the heartbeat only resolves missions with `status === 'IN_PROGRESS'`, so no production code ever transitioned missions to the active state.
- `saveGame` did not include `exploring` in the serialized payload, breaking offline mission resolution on reload.

## What was done

### 1. Wired heartbeat lifecycle into the game store
Added `heartbeat: HeartbeatInstance | null` to `GameStoreState` and `startHeartbeat` / `stopHeartbeat` actions. `startHeartbeat` is called from `initGame`, `login`, `register`, and `loadGame` so the 1-second interval loop runs for the entire session.

### 2. Converted `movePlayer` to queue-based exploration
`movePlayer` now:
- Validates bounds, energy, and previous exploration state exactly as before.
- For already-explored tiles, calls `finishMovement` immediately (instant travel).
- For unexplored tiles, calls `addMissionWithModifiers` to create an `EXPLORE_TIER_1` mission with `duration_seconds = currentWorldId × 30`.
- Stores the mission's compressed `end_time` and duration in the `exploring` state so the existing UI progress calculation continues to work.
- Removed the raw `setTimeout` callback.

### 3. Ensured missions are active when queued
Updated `addMission` and `addMissionWithModifiers` to immediately set `status: 'IN_PROGRESS'`. This allows the heartbeat to resolve them when `end_time` passes.

### 4. Implemented `EXPLORE_TIER_1` resolution callback
The heartbeat's `resolveMissionCallbacks.EXPLORE_TIER_1` handler:
- Reads the current `exploring` state (if any).
- Calls `finishMovement(targetX, targetY, tileKey, true)`.
- `finishMovement` performs tile discovery, neighbor fog-of-war rendering (`processTileDiscovery`), quest progress updates, energy drain, and the encounter roll.

### 5. Fixed offline catch-up
`processOfflineCatchUp` now:
- Transitions any loaded `PENDING` missions to `IN_PROGRESS` before the heartbeat tick.
- Uses the real `EXPLORE_TIER_1` callback (previously a no-op) so explorations and other mission types can resolve after long offline periods.

### 6. Persisted `exploring` state in save/load
Added `exploring: state.exploring` to the save payload and `exploring: data.exploring || null` to load so the heartbeat has the target coordinates when resolving catch-up missions.

## Files Modified

- `summoner-world/src/stores/game/types.ts` – Added `HeartbeatInstance` import and `heartbeat`, `startHeartbeat`, `stopHeartbeat` to store interfaces.
- `summoner-world/src/stores/game/modules/missionModule.ts` – Rewrote `movePlayer` to register queue missions; missions now start `IN_PROGRESS`; added `startHeartbeat` and `stopHeartbeat` actions; enabled real `EXPLORE_TIER_1` resolution in both live heartbeat and offline catch-up.
- `summoner-world/src/stores/game/modules/playerModule.ts` – `startHeartbeat()` called in `initGame`, `login`, `register`, and `loadGame`. Also added `exploring` to save/load serialization.
- `summoner-world/src/stores/gameStore.ts` – Added `heartbeat: null` to initial game state.

## Behavioral Changes

| Aspect | Before | After |
|--------|--------|-------|
| Movement into unknown tiles | `setTimeout` tied to closure over local vars | `EXPLORE_TIER_1` mission enqueued, resolved by heartbeat |
| Mission urgency | `PENDING` forever (never resolved) | `IN_PROGRESS` automatically when added |
| Heartbeat usage | Only during offline catch-up with empty callbacks | Live 1-second loop active for the whole session |
| Offline catch-up | `EXPLORE_TIER_1` was a no-op | Resolves missing explorations and updates player state |
