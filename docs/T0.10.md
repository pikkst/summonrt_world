# T0.10 – Add `.env.example` and Document dotenv Usage for Server

## Task Description
Add `.env.example` file and document dotenv usage for the server so developers can configure environment variables without hardcoding secrets or connection strings.

## Investigation

### Server Location
The Express server is located at `summoner-world/server/`.

### Findings
1. **dotenv is already a dependency** in `server/package.json` (`"dotenv": "^16.3.1"`).
2. **dotenv is already loaded** in `server/index.js` (line 1: `require('dotenv').config();`).
3. **Two environment variables are used** in the server:
   - `PORT` — HTTP listen port (fallback: `5000`)
   - `MONGODB_URI` — MongoDB connection string (fallback: `mongodb://localhost:27017/summoner_world`)
4. **No `.env.example` file existed** to document expected variables for new developers.
5. **No `.env` file existed** either (confirmed by glob search).

## Changes Made

### Created `.env.example`
Created `summoner-world/server/.env.example` with documented defaults:
- `PORT=5000`
- `MONGODB_URI=mongodb://localhost:27017/summoner_world`

### Usage
1. Copy `.env.example` to `.env`:
   ```bash
   cp .env.example .env
   ```
2. Edit `.env` with actual values (e.g., production MongoDB URI).
3. Start the server — `dotenv` automatically loads `.env` on line 1 of `index.js`.

### Security Note
- `.env` must NOT be committed to version control. This will be addressed in T0.11 (`.gitignore` entries).

## Verification
- Reviewed `server/index.js` — all `process.env` usages are now covered by `.env.example`.
- Verified dotenv package is installed and initialized before any `process.env` access.

## Task Status
✅ **Done** — `.env.example` added and dotenv usage documented.
