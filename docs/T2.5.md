# Task T2.5 – Implement Offline Catch-Up

## What was found

The game lacked offline catch-up logic. When a player closed the browser and reopened it later, missions that completed during the offline period were not resolved. The `heartbeat.ts` from T2.4 provided the `tick()` mechanism to detect expired missions, but it was only used during live play (1-second interval loop). There was no mechanism to:

- Record when the player logged out
- Persist active missions across sessions
- Batch-resolve expired missions on login/load

Additionally, `missonQueue.ts` had a latent bug where iterating over the missions array while removing elements could skip elements. The `playerModule.ts` `saveGame()`/`loadGame()` functions did not persist missions. The store's `missionActions` were missing a `removeMission` action required by the heartbeat's `resolveMission`.

## What was done

### 1. Added defensive copy in `heartbeat.ts`

The `tick()` function now iterates over a shallow copy of the missions array (`[...callbacks.getMissions()]` instead of the raw reference). This prevents element-skipping when `resolveMission` triggers `removeMission` during iteration.

### 2. Added `removeMission` to store mission actions

`src/stores/game/modules/missionModule.ts` now exports `removeMission(missionId)` which filters the mission out of the array entirely (distinct from `completeMission`/`failMission` which only change status).

### 3. Added `processOfflineCatchUp` store action

`processOfflineCatchUp(logoutTimestamp: number): number` creates a heartbeat instance with the current store as callbacks, calls `tick()` once to resolve all expired missions, logs the result to the player, and returns the count of resolved missions.

### 4. Added `lastLogoutTimestamp` to persistence layer

- `GameStoreState` gained `lastLogoutTimestamp?: number`
- `logout()` now saves `Date.now()` to both store state and `localStorage` key `summonerworld-last-logout`
- `saveGame()` persists `missions` and `lastLogoutTimestamp`
- `loadGame()` restores `missions` and `lastLogoutTimestamp`, then calls `processOfflineCatchUp` if missions exist

### 5. Integrated catch-up into `login()`

After the server login flow populates state, the function checks `localStorage` for:
- A saved game file with missions (merged into store)
- A stored logout timestamp

If missions exist after the merge, `processOfflineCatchUp` runs and the localStorage timestamp is cleared.

### 6. Added tests for batch offline resolution

`src/test/heartbeat.test.ts` gained three new test cases:
- **8h offline catch-up**: verifies that quick and long missions are resolved while a far-future mission is preserved
- **Completed mission skip**: verifies COMPLETED missions are not re-processed
- **PENDING mission skip**: verifies PENDING missions are not resolved prematurely
